## 卡牌目录分布

当前 `src/Core/Models/Cards/` 目录下的实际文件数量大致如下：

- `basic`：5
- `shine`：24
- `promisePile`：31
- `neutral`：11
- `globalMove`：3
- `relic`：7
- `strength`：2
- `token`：8
- `curses`：2
- `ancient`：1

这些数字本身就很有信息量：

- PromisePile（约定牌堆）是当前最重的主题目录
- Shine（闪耀）是第二重主题
- `token`、`relic`、`globalMove` 等目录说明 Karen 并不只是“普通主卡 + 少量机制卡”，而是已经有较多派生牌和特殊来源牌

卡池总入口：

- `src/Core/Models/CardPools/KarenCardPool.cs`

如果你想确认一张牌是否真的进入了角色池，最终还是要以 `KarenCardPool` 为准，而不是只看文件是否存在。

## 当前卡池的组织方式

从 `KarenCardPool.GenerateAllCards()` 可以看出，项目当前不是自动扫目录建卡池，而是手工逐个 `ModelDb.Card<T>()` 枚举所有卡牌。

这种方式的优点：

- 控制力强
- 不容易误把测试牌、临时牌放进正式卡池

代价：

- 新增卡后必须记得补到 `KarenCardPool`
- 删除或改名卡牌时也要同步维护这里

## 按机制粗分

如果按机制而不是目录来找，当前卡牌大致可以分成：

- 基础卡
  起始打防、开局骨架和少量通用牌。
- Shine（闪耀）卡
  直接带闪耀值，或围绕闪耀消耗、恢复、奖励展开。
- PromisePile（约定牌堆）卡
  与加入、抽取、切换、清空、自动打出相关。
- GlobalMove（全局移牌）联动卡
  依赖牌堆移动事件触发。
- Relic / DisableRelic（禁用遗物）卡
  与遗物或禁用遗物数量挂钩。
- Token / Option 卡
  战斗内生成、作为选项出现、通常不直接进正式卡池。
- Curse / Ancient 相关卡
  特殊来源、实验性或机制测试用途。

## 卡池与目录不完全等价

当前项目里有不少牌虽然在 `Cards/` 下存在，但它们的来源和用途并不相同：

- 有些是正式可获得牌
- 有些是战斗内生成牌
- 有些是选项牌
- 有些是测试 / 兼容 / 特殊来源牌

因此查卡时建议先分两步：

1. 先确认它是“存在于代码中的卡”
2. 再确认它是“正式进入 KarenCardPool 的卡”

## 常见检索方法

当你想找一个现成实现时，优先按以下顺序查：

1. 看 `KarenCardPool.cs`，确认卡是否真的进卡池。
2. 看目录名，先判断它属于哪种机制。
3. 搜基类覆写点，例如 `OnPlay`、`CanonicalVars`、`OnUpgrade`。
4. 搜关键系统钩子，例如 `OnAddedToPromisePile`、`OnShineExhausted`、`OnGlobalMove`。
5. 搜同名 Power、Relic、Patch 是否有联动。
