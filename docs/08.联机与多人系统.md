# 联机与多人系统

> 主题：联机模式下需要注意的流程与约束  
> 适用范围：原生 STS2 联机、选择流程同步、Karen 自定义系统的联机兼容点  
> 阅读建议：只有涉及联机、录像或同步问题时再深入本篇

---

## 原生联机架构

从当前代码和引用可以确认，STS2 联机侧至少涉及：

- `INetGameService`
- `PlayerChoiceSynchronizer`
- `NetGameType`
- `RunManager.Instance.NetService`

Karen 文档层最需要记住的是：

- 游戏主要同步的是“输入和选择”
- 不是直接同步整份运行时状态
- 因此任何自定义逻辑只要绕过原版选择链路，就很容易在联机或录像下出问题

## PlayerChoiceContext 的地位

`PlayerChoiceContext` 是 Karen 系统里非常核心的一个联机安全入口。

它至少出现在这些地方：

- 卡牌 `OnPlay(PlayerChoiceContext, CardPlay)`
- Shine 耗尽回调 `OnShineExhausted(...)`
- PromisePile 抽牌、选牌、切换手牌
- 自定义选卡界面 `CardSelectCmdEx`

从设计上说，它的意义是：

- 把玩家选择纳入可同步、可等待、可记录的上下文
- 避免你在自定义系统里偷偷做一个本地 UI 选择而联机另一端完全不知道

## 当前项目里和联机最相关的代码点

### `CardSelectCmdEx`

文件：`src/Core/Commands/CardSelectCmdEx.cs`

这份扩展命令说明了项目是怎么处理“自定义选卡界面但仍兼容联机”的：

- 先通过 `PlayerChoiceSynchronizer.ReserveChoiceId(player)` 预留选择 ID
- `SignalPlayerChoiceBegun(...)`
- 本地玩家做出选择后同步结果
- 远端玩家则等待 `WaitForRemoteChoice(...)`
- 最后 `SignalPlayerChoiceEnded()`

换句话说，自定义 UI 没问题，但必须把结果挂回原版选择同步链路里。

### PromisePile 命令

`PromisePileCmd` 内很多方法都显式要求：

- `PlayerChoiceContext ctx`

例如：

- `Draw(...)`
- `SelectedToHand(...)`
- `AddFromDiscard(...)`
- `AddFromDraw(...)`
- `AddFromHand(...)`
- `DiscardAll(...)`
- `SwitchHand(...)`
- `AutoPlayFromPromisePile(...)`

这说明 PromisePile 在设计上已经默认自己要兼容联机，而不是只为单机写的系统。

### Shine 系统

Shine 耗尽流程里会把 `PlayerChoiceContext` 缓存到卡牌，再在后续重定向流程中取出。

这点很重要，因为 Shine 并不是同步地在卡牌 `OnPlay()` 里就结束了，它后面还有：

- 结果牌堆重定向
- ShinePile 迁移
- 可能的后续回调

如果不把上下文跟着卡牌一起传过去，后面的选择类行为就会脱离联机上下文。

## 联机下最容易出问题的点

### 1. 本地独占的选卡 / 选项逻辑

如果你直接弹一个本地界面选牌，而不走：

- `PlayerChoiceContext`
- `PlayerChoiceSynchronizer`

那么：

- Host 和 Client 的结果会不一致
- Replay 也可能复现不出来

### 2. 自定义牌堆逻辑只改了显示没改状态

PromisePile、Void 模式、额外奖励牌、生成牌都属于这种高风险区：

- 本地看起来对了
- 但如果真实状态没被同步链路正确记录，联机会出问题

### 3. 修改打牌次数或牌堆迁移

ExtraReplay 和 GlobalMove 都会影响原始动作序列。

只要改了：

- 打牌次数
- 自动打出
- 结果牌堆
- 生成牌顺序

就要格外注意联机和 Replay 的一致性。

## 当前文档层面的结论

Karen 自定义系统里，最联机敏感的几块是：

- 选牌 / 选项
- PromisePile
- Shine 耗尽链路
- 自动打出
- 额外奖励

这些位置都不适合写“只在本地运行就行”的快捷逻辑。

## 排查顺序建议

如果某个功能在联机中异常，建议按下面顺序排查：

1. 问题是显示不同步，还是状态不同步。
2. 有没有绕过 `PlayerChoiceContext`。
3. 有没有绕过 `PlayerChoiceSynchronizer`。
4. 是否引入了本地专属随机、顺序依赖或只在 UI 上生效的逻辑。
5. 是否改了牌堆移动 / 播放次数 / 生成牌顺序却没有同步看待。

## 联机特效显示规则

自定义特效需要先判断它属于“公开战斗信息”还是“本地操作反馈”。公开战斗信息可以给所有玩家显示；依赖本地手牌、鼠标悬停、全屏演出、专属音乐的内容，应只给对应玩家本地显示。

通用判断：

- 角色身上的公开持续状态、敌人身上的命中特效、公开资源数量提示：所有玩家可见。
- 手牌位置、牌堆节点、鼠标 Hover、选择预览、全屏 Cutscene、专属 BGM：仅本地玩家可见或可听。
- 只限制视觉 / 音频，不应限制真实逻辑结算。联机状态仍要走 `PlayerChoiceContext`、`PromisePileCmd`、`PowerCmd` 等同步链路。
- 本地判断优先使用 `LocalContext.IsMe(player)` 或 `LocalContext.IsMine(card)`。

当前 Karen 特效策略：

| 来源 | 特效 | 联机显示策略 | 说明 |
|---|---|---|---|
| `KarenForm` | 角色身上的 Form 视觉特效 | 仅拥有者本地显示 | 已在 `KarenFormVfxManager.Start/Stop` 用 `LocalContext.IsMe(player)` 过滤。 |
| `KarenForm` | Form 专属 BGM | 仅拥有者本地播放 | 已在 `KarenFormMusicManager.PlayLoop/Stop/StopForCutscene` 用本地玩家过滤。 |
| `KarenLastWord` | 全屏视频 | 仅出牌者本地显示 | 这是出牌者专属 Cutscene。 |
| `KarenLastWord` | T 字演出、专属音效 | 所有玩家可见 / 可听 | 非出牌者不播放视频，但会显示 T 字、播放音效并等待相同演出时长。 |
| `KarenStarGuide` | 手牌 / 牌堆飞星演出、手牌节点移除 | 仅出牌者本地显示 | 逻辑移牌仍执行 `PromisePileCmd.Add` 同步。 |
| `KarenCarryingGuilt` | Hover 聚光灯预览 | 仅卡牌拥有者本地显示 | Hover 是本地输入反馈，不广播给其他玩家。 |
| `KarenCarryingGuilt` | 实际出牌时的舞台灯命中特效 | 所有玩家可见 | 命中目标和伤害是公开战斗结果。 |
| `KarenBurn` | Burn 模式持续特效 | 所有玩家可见 | 公开的角色状态表现。 |
| `KarenPastAndFuture` | Past / Future 环和 Pulse | 所有玩家可见 | 公开的角色状态与触发反馈。 |
| PromisePile 星星 | 约定牌堆数量星星 | 所有玩家可见 | 公开资源数量提示。 |

新增特效时，建议先把入口分成两类：

- `PlayPublic...`：公开战斗表现，不做本地过滤。
- `PlayLocal...`：本地 UI / 音频 / Cutscene 表现，入口处先判断 `LocalContext.IsMe(...)`。
