# Karen核心系统

> 主题：Karen 专属机制的系统说明  
> 适用范围：Shine（闪耀）、PromisePile（约定牌堆）、GlobalMove（全局移牌）、ExtraReplay（额外重播）、DisableRelic（禁用遗物）及相关扩展  
> 阅读建议：系统改动优先先看本篇，再回到 `03` 和 `09`

---

## 包含哪些系统

Karen 当前最核心的自定义系统包括：

- Shine（闪耀）
- PromisePile（约定牌堆）
- GlobalMove（全局移牌）
- ExtraReplay（额外重播）
- DisableRelic（禁用遗物）
- 奖励扩展
- 存档扩展
- 角色资源与能量相关表现

这些系统大多不是各管各的，而是会在打牌、牌堆移动、奖励、读档时互相接力。

## Shine（闪耀）

核心文件：

- `src/Core/ShineSystem/ShineExtension.cs`
- `src/Core/ShineSystem/Patches/ShinePatch.cs`

### 数据结构

Shine 当前通过 `SpireField<CardModel, ...>` 附着在任意卡牌上，主要字段有：

- 当前闪耀值：`_shineCurrent`
- 最大闪耀值：`_shineMax`
- 是否在打出后强制进入 ShinePile：`_enterShinePileAfterPlay`

这代表：

- Shine 不是某种固定卡牌子类专属属性
- 任意卡只要初始化了 Shine 字段，就能被系统当作闪耀卡处理

### 主要 API

- `IsShineCard()`
- `HasShine()`
- `GetShineValue()`
- `GetShineValueRounded()`
- `GetShineMaxValue()`
- `AddShineMax(...)`
- `SetShineCurrent(...)`
- `SetShineMax(...)`
- `DecreaseShine()`
- `RestoreShineToMax()`
- `RemoveShine()`
- `SetEnterShinePileAfterPlay(...)`

### 流程接线

`ShinePatch` 当前主要做了三件事：

1. 在 `CardModel.OnPlayWrapper` 前缀里减少闪耀值
2. 在 `Hook.ModifyCardPlayResultPileTypeAndPosition` 中改写结果牌堆
3. 拦截 `CardPileCmd.Add(...)`，接管应进入 ShineDepletePile 的卡牌

### 额外细节

- 如果卡牌不是 `Eternal`，且有 Shine，则打牌时会递减
- 减值后会同步到 `deckVersion`
  这样后续从牌堆抽出的对应克隆能保留正确 Shine 状态
- Shine 系统给 `AbstractModel.MutableClone` 补了克隆逻辑
  用来复制 Shine 当前值和最大值

### 设计意图

从现有代码看，Shine 的定位不是简单“多段使用次数”，而是：

- 一种可见的卡牌寿命
- 一条能重写结果牌堆流向的规则
- 一套与动画、存档和复制链路绑定的系统

## PromisePile（约定牌堆）

核心文件：

- `src/Core/Commands/PromisePileCmd.cs`
- `src/Core/PromisePileSystem/PromisePileManager.cs`
- `src/Core/Models/Powers/KarenPromisePilePower.cs`

### 数据结构

`PromisePileManager` 用：

- `SpireField<PlayerCombatState, CardPile>`

把约定牌堆附着在玩家战斗状态上。

这个设计说明：

- 每个玩家一份独立的 PromisePile
- 它只在战斗内生效
- 它不是原版 `PileType` 直接管理的普通常规牌堆

### PromisePileManager 负责什么

当前它至少负责：

- 获取当前玩家的 PromisePile
- 判断卡牌是否在 PromisePile 中
- 判断玩家是否处于特殊模式
- 进入 PromisePile 模式
- 向 PromisePile 加牌
- 从 PromisePile 抽牌
- 从其他牌堆选牌放入 PromisePile
- 清空 PromisePile
- 更新 PromisePile 对应的 Power 和星星 VFX 数量
- 打开 PromisePile 查看界面

### 主要模式

当前 `KarenPromisePilePower` 里定义了 `PromisePileMode`：

- `Void`
- `InfiniteReinforcement`
- `Burn`

它们的含义大致是：

- `Void`
  把原本 PromisePile 的很多交互重定向到抽牌堆
- `InfiniteReinforcement`
  让系统持续把 PromisePile 维持在特定填充状态
- `Burn`
  让从 PromisePile 取出的牌带额外升级 / 消耗效果

### PromisePileCmd 负责什么

`PromisePileCmd` 是卡牌层最常直接使用的入口。

它内部已经处理了大量模式分支和上下文：

- `Add`
- `Draw`
- `DrawToFullHand`
- `SelectedToHand`
- `AddFromHand`
- `AddFromDiscard`
- `AddFromDraw`
- `MoveAllToDrawPile`
- `DiscardAll`
- `DiscardCards`
- `SwitchHand`
- `AutoPlayFromPromisePile`
- `EnterMode`

也就是说，卡牌效果层通常不该直接操作 `PromisePileManager` 的内部卡堆，而应该先走 `PromisePileCmd`。

### 视图与交互

PromisePile 不是只有规则层，当前还包含：

- 查看界面：
  `ShowScreen(...)`
- 牌堆数量和星星显示：
  `KarenPromiseVfxStarManager.UpdatePromisePileStarCount(...)`
- Void 模式抽牌堆图标覆盖：
  `DrawPileIconCmd.Override(...)`

说明 PromisePile 已经是一个完整玩法子系统，而不是单纯的后台容器。

## GlobalMove（全局移牌）

核心文件：

- `src/Core/GlobalMoveSystem/GlobalMoveSystem.cs`
- `src/Core/GlobalMoveSystem/Patches/GlobalMovePatch.cs`

### 它解决的问题

原版流程中，卡牌常常会先移动到 `Play`，再移动到最终牌堆。

如果直接监听每次物理移动，会有两个问题：

- 卡牌收到太多“中转噪音”事件
- 很难表达真正的玩法语义

### 当前实现

`GlobalMovePatch` 会：

- 在 `Hook.AfterCardChangedPiles` 之后介入
- 如果牌进入 `Play`，先缓存它打出前的来源牌堆
- 如果后续又从 `Play` 离开，则把两段移动拼接成一次更稳定的移动语义

最后由：

- `GlobalMoveSystem.Trigger(card, oldPile, newPile, source)`

去调用卡牌的：

- `OnGlobalMove(...)`

### 当前边界

- 模板卡不处理
- 某些 `None / Deck` 目标会被过滤
- `Play` 牌堆是中转语义，不直接暴露给卡牌逻辑

## ExtraReplay（额外重播）

核心文件：

- `src/Core/ExtraReplaySystem/ExtraReplayExtension.cs`
- `src/Core/ExtraReplaySystem/Patches/ExtraReplayPatch.cs`

### 数据结构

用一个 `SpireField<CardModel, int>` 记录：

- `_extraReplayCountForNextPlay`

### API

- `GetExtraReplayCountForNextPlay()`
- `SetExtraReplayCountForNextPlay(...)`
- `AddExtraReplayCountForNextPlay(...)`

### 生效方式

`ExtraReplayPatch` 给 `Hook.ModifyCardPlayCount` 打了 `Postfix`：

- 读取卡牌上的额外重播次数
- 累加到本次实际播放次数
- 使用后立即清零

这说明 ExtraReplay 是“下次打出时消耗掉”的一次性状态，而不是常驻属性。

## DisableRelic（禁用遗物）

核心文件：

- `src/Core/DisableRelicSystem/DisableRelicVar.cs`
- `src/Core/Models/Cards/KarenDisableRelicBaseCardModel.cs`
- `src/Core/DisableRelicSystem/Patches/DisableRelicViewPatch.cs`

### 当前结构

这一套系统的核心并不复杂：

- `DisableRelicVar` 提供文本和数值层的动态变量
- `KarenDisableRelicBaseCardModel` 负责可打出性判断
- 视图补丁负责把数值正确显示到界面

### 实际玩法含义

对这类牌来说，“是否可打”和“描述里显示多少需求”是一体的，不能只做其中一半。

## 奖励扩展

当前奖励逻辑已经能看到自定义入口：

- `ExtraRewardCmd.AddShineCardReward(...)`
- `ExtraRewardCmd.AddRelicReward(...)`

其中：

- Shine 卡奖励会从 `ShineManager.GetAllShineCards()` 加上额外指定卡里随机取牌
- 战斗房间中直接用 `CombatRoom.AddExtraReward(...)`
- 非战斗场景则走 `RewardsCmd.OfferCustom(...)`

这说明奖励扩展已经兼顾了“房间内额外奖励”和“直接弹出奖励”两类场景。

## 存档扩展

当前存档结构主要在：

- `src/Core/SaveSystem/KarenRunSaveData.cs`
- `src/Core/SaveSystem/KarenModSaveBuffer.cs`
- `src/Core/SaveSystem/Patches/RunSaveManager_Patches.cs`

从代码能确认：

- Karen 自定义数据写入 `karen_mod_data`
- 当前至少保存：
  - 每名玩家的 Shine 数据
  - 每名玩家的 ShinePile 数据
- 读取时先缓存在 `KarenModSaveBuffer`
- 等 `RunManager.SetUpSavedSinglePlayer / SetUpSavedMultiPlayer` 完成后再恢复

这是一个典型的“两阶段恢复”设计：

- 第一阶段只提取 JSON
- 第二阶段等运行时对象就绪后再映射回卡牌实例

## 系统改动时的检查点

修改任一 Karen 核心系统时，建议至少检查：

1. 是否影响卡牌基类和现有 Hook（钩子）。
2. 是否影响结果牌堆或牌堆可视化。
3. 是否会破坏联机和 Replay 的一致性。
4. 是否需要补克隆、生成牌或 deckVersion 同步。
5. 是否需要补存档、读档或版本迁移。
6. 是否要同步更新提示文本、HoverTip 和视图补丁。
