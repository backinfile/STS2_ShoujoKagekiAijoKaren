# STS2 单机 vs 联机架构分析

> 源码版本：v0.99.1 (commit `7ac1f450`)

---

## 1. 架构总览：单机 vs 联机的根本区别

### `INetGameService` 抽象

游戏通过 `INetGameService` 接口在单机/联机之间切换，共 4 种实现：

| 类型 | 描述 |
|---|---|
| `NetGameType.Singleplayer` | `NetSingleplayerGameService`（本地，无网络） |
| `NetGameType.Host` | `NetHostGameService`（作为权威服务器） |
| `NetGameType.Client` | `NetClientGameService`（连接到 Host） |
| `NetGameType.Replay` | `NetReplayGameService`（确定性回放） |

`RunManager` 是持有所有同步器的核心对象。几乎所有联机逻辑都通过以下判断门控：
```csharp
if (_netService.Type != NetGameType.Singleplayer) { ... }
```

**核心架构：锁步确定性同步（Lockstep Deterministic）**
所有玩家运行相同的游戏逻辑，网络只传输输入，不传输状态差值。状态分歧通过校验和（Checksum）检测而非状态推送修复。

### 联机专属系统（单机下为 null 或 no-op）

| 系统 | 单机 | 联机 |
|---|---|---|
| `RunLobby` | null | 激活 |
| `CombatStateSynchronizer` | IsDisabled = true | 激活 |
| `ChecksumTracker` | 不发送消息 | 发送/接收校验和 |
| `ActionQueueSynchronizer` | 直接入队 | 请求/确认往返 |
| `PlayerChoiceSynchronizer` | 不存在 | 异步等待远程 |
| `MapSelectionSynchronizer` | 不存在 | 投票收集 + Host 决定 |
| `RewardSynchronizer` | 不存在 | 广播 + 应用到远程玩家 |

### 其他关键差异

```csharp
// 联机时，房间生成传入 isMultiplayer = true，可调整房间内容和权重
act.GenerateRooms(State.Rng.UpFront, State.UnlockState, State.Players.Count > 1);

// 联机时，教程不触发
// TryGetRoomTypeForTutorial() 在 Players.Count > 1 时直接返回 false

// 联机时，进度解锁跳过
public bool ShouldIgnoreUnlocks => IsInProgress && !IsSinglePlayerOrFakeMultiplayer;

// Abandon 流程不同
if (NetService.Type == NetGameType.Singleplayer)
    TaskHelper.RunSafely(AbandonInternal());
else
    RunLobby.AbandonRun(); // 先广播 RunAbandonedMessage 给所有客户端
```

## 2. 联机网络实现

### 传输层：Steam P2P

使用 Steam `SteamNetworkingSockets` P2P API，封装为 `SteamHost` 和 `SteamClient`。
备用传输层：`ENetClientConnectionInitializer`（ENet）。

**Host (`SteamHost.cs`)：**
- 调用 `SteamMatchmaking.CreateLobby(k_ELobbyTypeFriendsOnly, maxPlayers)` 创建大厅
- 用 `SteamNetworkingSockets.CreateListenSocketP2P(0, ...)` 建立监听 socket
- 连接超时：20,000ms
- 只接受 Steam 大厅成员（通过 `IsInLobby()` 校验）
- 开局后调用 `SetHostIsClosed(true)` 将大厅类型改为 `k_ELobbyTypePrivate`，阻止新玩家加入

**Client (`SteamClient.cs`)：**
- 加入 Steam 大厅后获取房主 ID，再用 `SteamNetworkingSockets.ConnectP2P` 建立连接
- 支持两种加入方式：通过好友 SteamID（`FromPlayer`）或直接大厅 ID（`FromLobby`）

**消息协议：** 所有消息实现 `IPacketSerializable`（自定义二进制协议，`PacketWriter`/`PacketReader`）。
传输模式按消息类型声明：
- `NetTransferMode.Reliable` — 游戏状态、奖励、动作
- `NetTransferMode.Unreliable` — 光标/悬停数据（`PeerInputMessage`）

### 大厅握手协议

#### 新局 (`StartRunLobby`)

```
1. Host 创建 Steam 大厅和 StartRunLobby
2. Client 连接 → Host 发送 InitialGameInfoMessage (sessionState=InLobby)
3. Client 发送 ClientLobbyJoinRequestMessage（携带 unlockState + maxAscensionUnlocked）
4. Host 验证人数，分配槽位，回复 ClientLobbyJoinResponseMessage
   （包含当前所有玩家、种子、天赋等级、修改器）
5. Host 广播 PlayerJoinedMessage 给其他所有玩家
6. 握手超时：10,000ms
7. 所有玩家就绪 → Host 选定种子 → 广播 LobbyBeginRunMessage → 双方调用 LobbyListener.BeginRun()
```

#### 读档局 (`LoadRunLobby`)

```
1. Host 创建 LoadRunLobby（持有 SerializableRun）
2. Client 连接 → Host 发送 InitialGameInfoMessage (sessionState=InLoadedLobby)
3. Client 发送 ClientLoadJoinRequestMessage
4. Host 通过 NetId 验证玩家存在于存档，回复 ClientLoadJoinResponseMessage
   （包含完整序列化 Run + 已连接玩家列表）
5. 广播 PlayerReconnectedMessage 给其他人
6. 所有玩家就绪 → Host 广播 LobbyBeginLoadedRunMessage → BeginRun()
```

#### 战斗中重连 (`RunLobby.HandleClientRejoinRequestMessage`)

```
1. 客户端发送 ClientRejoinRequestMessage
2. Host 调用 RunManager.GetRejoinMessage() → 创建 ClientRejoinResponseMessage
   （包含完整当前 SerializableRun + NetFullCombatState）
3. 广播 PlayerRejoinedMessage 给其他玩家
```

### 游戏内同步机制

#### ActionQueueSynchronizer（动作队列权威模式）

这是锁步模型的核心。**只有 Host 才能真正入队 Action：**

```
Client 想打一张牌：
  Client → RequestEnqueueActionMessage → Host
  Host 验证，调用 EnqueueAction()
  Host → ActionEnqueuedMessage → 广播给所有 Client
  所有节点（包括 Host 自己）同时执行该动作
```

三种动作子类型：
- 普通动作：`RequestEnqueueActionMessage` / `ActionEnqueuedMessage`
- Hook 动作（遗物/卡牌异步触发）：`RequestEnqueueHookActionMessage` / `HookActionEnqueuedMessage`
- 玩家选择恢复：`RequestResumeActionAfterPlayerChoiceMessage` / `ResumeActionAfterPlayerChoiceMessage`

#### PlayerChoiceSynchronizer（玩家选择同步）

当战斗动作需要玩家做出选择（如指定目标、丢弃哪张牌）：
- 每个玩家每个槽位分配 `choiceId`（顺序 uint 计数器）
- 本地玩家：`SyncLocalChoice()` → 广播 `PlayerChoiceMessage`
- 远程玩家：`WaitForRemoteChoice()` 返回 `Task<PlayerChoiceResult>`（由 `TaskCompletionSource` 支持），等待对应消息到达时完成
- 支持预到达缓冲：消息若先于游戏准备好到达，暂存后立即完成

#### CombatStateSynchronizer（房间切换前状态同步屏障）

在进入每个新房间前，执行全量状态同步：
```
1. 每个节点广播 SyncPlayerDataMessage（序列化的 SerializablePlayer）
2. Host 额外广播 SyncRngMessage（完整 SerializableRunRngSet + SerializableRelicGrabBag）
3. WaitForSync() 异步阻塞，等待所有玩家消息到齐
4. 每个节点用远程玩家数据更新对应 player 对象
5. 非 Host 节点用 Host 的 RNG 状态覆盖本地状态（Host 的 RNG 有权威性）
```

触发时机：`RunManager.EnterMapPointInternal` 和 `EnterRoomWithoutExitingCurrentRoom`（每次进入新房间）。

目的：修复跨房间期间各玩家独立的遗物/卡牌效果导致的状态漂移，重置共享 RNG，保证后续抽随机的确定性。

#### ChecksumTracker（状态分歧检测）

每次战斗中 `GameAction` 执行后，触发 `SendPostActionChecksum()`：
1. 将 `NetFullCombatState` 序列化后用 `XxHash32` 计算 32 位校验和
2. Client 发送 `ChecksumDataMessage` 给 Host
3. Host 对比本地同序号的校验和
4. 不一致时：Host 发送 `StateDivergenceMessage`（含 Host 完整状态快照）给 Client
5. Client 收到分歧通知 → 触发 `StateDiverged` 事件 → 写入回放日志 → Abandon 游戏
6. 上报 Sentry

维护最近 20 个校验和的滚动窗口，处理乱序到达。

#### MapSelectionSynchronizer（地图移动投票）

联机中地图移动通过投票决定：
- 每个玩家调用 `PlayerVotedForMapCoord()`，存储其 `MapVote`（坐标 + 生成计数）
- 所有玩家投票后，**只有 Host** 调用 `MoveToMapCoord()`
- Host 使用以 `RunState.Rng.Seed` 为种子的 `_multiplayerMapPointSelection` RNG 从投票中选择目标
- 最终 `MoveToMapCoordAction` 通过 `ActionQueueSynchronizer` 入队（经正常广播路径分发给所有客户端）

#### RewardSynchronizer（战斗后奖励同步）

每个玩家独立做出奖励选择（卡牌/遗物/药水/金币），然后广播结果：
- 本地：`SyncLocalObtainedCard(card)` → 发送 `RewardObtainedMessage`（`ShouldBroadcast=true`）
- 收到远程玩家消息后：应用奖励到对应远程玩家状态
- 在战斗期间到达的消息缓冲到 `_bufferedMessages`，战斗结束时统一处理
- 消息携带 `RunLocation` 字段，确保奖励被应用到正确的地图历史条目

#### PeerInputSynchronizer（视觉光标同步）

纯视觉，不影响游戏状态：
- 每帧发送 `PeerInputMessage`（不可靠 UDP）：鼠标位置（16bit 量化）、屏幕类型、悬停对象、鼠标按下状态、手柄信息
- 用于显示其他玩家光标的视觉覆盖层
- 玩家断线时 `OnPlayerDisconnected()` 清除其输入状态

## 3. 存档系统：单机 vs 联机差异

### 两个独立存档文件

```
current_run.save        — 单机存档
current_run_mp.save     — 联机存档
```

两者位于同一 `<profile_dir>/saves/` 目录下，格式完全相同（`SerializableRun` JSON），但互相独立。
**好处**：允许单机和联机各自有进行中的存档，互不干扰；防止不完整的联机会话损坏单机存档。

### 谁写存档

```csharp
// RunSaveManager.SaveRun()
if (netService.Type != Singleplayer && netService.Type != Host)
    return; // Client 永远不写存档
```

**只有单机或 Host 写存档，Client 不写。** Host 是唯一的真相来源。

路径选择：
```csharp
string savePath = netService.Type.IsMultiplayer()
    ? CurrentMultiplayerRunSavePath   // current_run_mp.save
    : CurrentRunSavePath;             // current_run.save
```

### 存档时机

存档在 `RunManager.EnterMapPointInternal()` 中触发（每次移动到新地图坐标）。
从存档中恢复当前房间时 `saveGame: false`，跳过写入（避免重复存档）。

### 联机存档内容

存档包含**所有玩家**的完整状态：
```csharp
Players = State.Players.Select(p => p.ToSerializable())
```
每个玩家的牌组、遗物、HP、金币、RNG 种子全部保存在 Host 拥有的单一文件中。

### 联机存档的规范化处理

`RunManager.CanonicalizeSave()` 对存档进行规范化处理：
- 验证 `localPlayerId` 存在于存档的玩家列表中
- 通过 `RunState.FromSerializable()` 再重新序列化，清理不应跨会话持久化的数据
- 用 `PacketWriter.Write(serializableRun)` 保证二进制表示的一致性
- 在 `RunSaveManager.LoadAndCanonicalizeMultiplayerRunSave()` 加载联机存档时调用
- 若规范化失败，存档被重命名为损坏文件（`RenameBrokenMultiplayerRunSave()`）

### 游戏结束时的存档清理

```csharp
if (NetService.Type == NetGameType.Singleplayer)
    SaveManager.Instance.DeleteCurrentRun();
else if (NetService.Type == NetGameType.Host)
    SaveManager.Instance.DeleteCurrentMultiplayerRun();
// Client：不需要清理，从未拥有存档
```

### 联机专属运行状态：`MultiplayerScalingModel`

`RunState.CreateShared()` 始终初始化 `MultiplayerScalingModel`：
```csharp
runState.MultiplayerScalingModel = (MultiplayerScalingModel)ModelDb.Singleton<MultiplayerScalingModel>().MutableClone();
runState.MultiplayerScalingModel.Initialize(runState);
```
该模型根据玩家数量缩放敌人 HP/伤害，参与 `IterateHookListeners()` 修改战斗属性。
**不单独序列化**，每次加载时根据 `Players.Count` 重新初始化。

## 4. 关键设计决策总结

| 决策 | 理由 |
|---|---|
| 锁步确定性模拟 | 所有节点运行相同逻辑，只传输输入。分歧用校验和检测，而非状态推送修复 |
| Host 作为动作排序权威 | 防止竞态（两名玩家同时打牌）。Client 请求 → Host 序列化 → 广播确认顺序给所有人 |
| 房间切换前全量状态同步屏障 | 房间间的独立遗物/卡牌效果可能导致漂移，在房间边界同步使所有节点对齐，无需传输每个微小变化 |
| Host 单一持有存档 | 最简一致性模型：单文件单写入方。Client 断线重连时从 Host 获取完整存档 |
| Steam P2P 大厅作为会话原语 | 大厅成员资格是连接授权的门槛，非成员在传输层被拒绝，无需游戏协议层鉴权 |
| 单机/联机存档文件分离 | 允许两种模式同时各自有进行中的存档，互不干扰 |

# STS2 联机架构分析

> 来源：`docs/Multiplayer_Architecture.md`，源码版本 v0.99.1

## 核心模型：锁步确定性同步（Lockstep Deterministic）

所有玩家运行相同的游戏逻辑，网络只传输输入，不传输状态差值。状态分歧通过 ChecksumTracker 检测。

## INetGameService 四种实现

| 类型 | 描述 |
|---|---|
| `NetGameType.Singleplayer` | 本地无网络 |
| `NetGameType.Host` | 权威服务器 |
| `NetGameType.Client` | 连接到 Host |
| `NetGameType.Replay` | 确定性回放 |

## 关键判断属性

```csharp
public bool IsSinglePlayerOrFakeMultiplayer => IsInProgress && NetService.Type == NetGameType.Singleplayer;
public bool ShouldIgnoreUnlocks => IsInProgress && !IsSinglePlayerOrFakeMultiplayer;
```

## 联机专属系统（单机下 null/no-op）

- `RunLobby`：联机大厅管理，单机为 null
- `CombatStateSynchronizer`：房间切换前全量状态同步屏障
- `ChecksumTracker`：每次 GameAction 后校验和比对
- `ActionQueueSynchronizer`：Host 作为动作排序权威（Client 请求 → Host 入队 → 广播）
- `PlayerChoiceSynchronizer`：异步等待远程玩家选择（TaskCompletionSource）
- `MapSelectionSynchronizer`：投票 + Host 用 RNG 决定最终目标
- `RewardSynchronizer`：奖励各自选择后广播同步

## 传输层

- Steam P2P（`SteamNetworkingSockets`）为主，ENet 为备
- 消息协议：自定义二进制（`PacketWriter`/`PacketReader`，实现 `IPacketSerializable`）
- Reliable（游戏状态/动作）+ Unreliable（光标位置）

## 存档差异

| 项目 | 单机 | 联机 |
|---|---|---|
| 文件名 | `current_run.save` | `current_run_mp.save` |
| 写入方 | 本地 | 只有 Host 写，Client 不写 |
| 存档内容 | 单玩家状态 | 所有玩家完整状态 |
| 加载时 | 直接读取 | Host 规范化后再加载；客户端从 Host 接收完整存档 |
| 结束清理 | `DeleteCurrentRun()` | Host 调 `DeleteCurrentMultiplayerRun()`，Client 无操作 |

## Mod 开发注意事项

- Shine 系统存档恢复 Postfix 同时 Hook 了 `SetUpSavedSinglePlayer` 和 `SetUpSavedMultiPlayer`，联机读档也能正确恢复
- `MultiplayerScalingModel` 不序列化，每次根据 `Players.Count` 重新初始化
- 联机时 `ShouldIgnoreUnlocks = true`，不要在联机中依赖解锁状态
- 房间生成时传入 `isMultiplayer = Players.Count > 1`，可用于自定义房间内容
