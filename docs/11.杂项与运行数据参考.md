## PlayerChoiceContext

`PlayerChoiceContext` 贯穿：

- 出牌流程
- Hook（钩子）流程
- 选牌与玩家交互
- 联机同步相关选择

当前项目里它出现的地方非常多，例如：

- 卡牌 `OnPlay(...)`
- Shine 耗尽回调
- PromisePile 抽牌、选牌、换手
- 自定义选卡界面

从架构上看，它更像一张“操作上下文票据”：

- 让后续动作知道这次行为是谁触发的
- 让联机和录像系统能接住这次选择
- 让异步链路里的后续步骤仍然能回到同一条操作上下文

写 Mod 时通常不需要自己新建它，而是沿用系统透传下来的 `choiceContext`。

## 选择同步相关对象

如果一个功能涉及选牌、选项或远端等待，通常还会和这些对象一起出现：

- `PlayerChoiceSynchronizer`
- `PlayerChoiceResult`
- `ReserveChoiceId(...)`
- `WaitForRemoteChoice(...)`

这些对象在 `CardSelectCmdEx` 里已经有比较完整的用法示例。

## RunHistory JSON

项目内保留了 `RunHistory` 输出格式参考，用于：

- 对局分析
- JSON 字段对照
- 运行结果上传或本地分析

虽然当前主干代码里和 `RunHistory` 直接相关的逻辑不算多，但它仍然是调试战斗流程和联机复盘时很有价值的参考。

## Karen 自定义存档数据

除了原版运行数据，Karen 还额外往存档里塞了：

- `karen_mod_data`

当前结构定义在：

- `KarenRunSaveData`

已知字段包括：

- `schema_version`
- `player_shine_data`
- `player_disposed_pile_data`

这些数据主要用于恢复：

- 每名玩家的 Shine 状态
- 每名玩家的 ShinePile 状态

## KarenModSaveBuffer 的用途

`KarenModSaveBuffer` 是一个过渡缓冲区。

它解决的问题是：

- 读取 JSON 时，运行时牌组实例还没完全就绪
- 不能直接把 Shine 数据立刻恢复回卡牌对象

所以当前流程是：

1. 读取时先 `Store(...)`
2. 等 `RunManager.SetUpSavedSinglePlayer / MultiPlayer` 完成
3. 再 `Consume()` 并恢复到当前运行时对象

这类两阶段恢复模式，后续如果再加别的运行时系统，也很值得复用。
