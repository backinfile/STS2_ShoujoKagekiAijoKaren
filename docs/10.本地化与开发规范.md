# 本地化与开发规范

> 主题：文本、本地化、资源命名与通用开发约定  
> 适用范围：描述文本、关键字、资源命名、通用实现习惯  
> 阅读建议：新增内容前快速过一遍，能减少返工

---

## 本地化结构

当前项目文本并不只存在于“卡牌描述”里，还分散在这些位置：

- 卡牌描述文本
- `card_keywords` 关键字提示
- HoverTip 标题与描述
- 牌堆 / 模式 / 系统说明文本
- 视图和界面用 LocString

从 `CardHoverTipsPatch` 和 `CardKeywordInlinePatch` 可以确认：

- 关键字系统已经深度参与卡牌展示
- 一些提示不是写死在卡牌类里，而是通过标签和本地化表动态生成

## 文本格式

当前项目常见的动态文本写法包括：

- `{Damage:diff()}`
- `{Block:diff()}`
- `{Cards:diff()}`
- `{Energy:energyIcons()}`

这些写法要求：

- 对应 `DynamicVar` 必须真的存在
- 升级差值要能被正确计算
- 文本、数值、升级逻辑三者必须一致

## DynamicVar 文本约定

当前文档和代码里已经固定使用：

- `DynamicVar（动态变量）`

常见变量类型：

- `DamageVar`
- `BlockVar`
- `CardsVar`
- `PowerVar<T>`
- `EnergyVar`
- `DisableRelicVar`

如果卡牌描述里用了动态变量，但 `CanonicalVars` 没配完整，通常会直接导致：

- 文本显示不对
- 差值显示异常
- 视图补丁取不到对应数值

## HoverTip 与关键字约定

`CardHoverTipsPatch` 的现状说明了一条重要规范：

- 不要只把一个机制写进卡牌描述
- 还要考虑它是否需要成为统一关键字

当前已接线的 Karen 关键字至少包括：

- Shine（闪耀）
- PromisePile（约定牌堆）
- TmpStrength
- DisableRelic（禁用遗物）
- TokenCard

如果你新增一个系统概念，建议同时判断是否需要：

1. 新标签
2. 新 HoverTip
3. 新 `card_keywords` 文本

## 常用术语写法

正文中建议统一使用以下写法：

- Shine（闪耀）
- PromisePile（约定牌堆）
- GlobalMove（全局移牌）
- ExtraReplay（额外重播）
- DisableRelic（禁用遗物）
- DynamicVar（动态变量）
- Power（能力）
- Patch（补丁）
- Hook（钩子）

建议正文保持“英文标识 + 中文说明”的写法，不要在不同文档里来回换叫法。

## 开发实现约定

从当前代码结构看，建议遵循这些习惯：

- 新卡默认优先使用 `KarenBaseCardModel`
- 需要打牌条件控制时，再考虑 `KarenDisableRelicBaseCardModel`
- 系统扩展优先复用已有命令类和补丁入口
- 涉及联机选择时必须沿用 `PlayerChoiceContext`
- 描述里提到的系统概念，尽量让标签、文本、HoverTip 一起跟上

## 资源命名与目录映射

当前项目结构倾向于让资源和代码目录一一对应。

建议保持这些映射关系稳定：

- 卡牌类名 ↔ 卡牌资源名
- 角色 ID ↔ 角色颜色 / 能量颜色名
- 系统名 ↔ 关键字 / 标签 / 本地化条目

例如：

- `Karen.energyColorName = "karen"`
- `KarenCardPool.Title => "karen"`

这类命名稳定后，后面补资源和补本地化都会轻松很多。

## 修改文本时要同步检查什么

每次改描述文本，建议顺手检查：

1. `CanonicalVars` 是否同步更新。
2. HoverTip 是否仍然准确。
3. 行内关键字显示是否会出排版问题。
4. 中文和英文版本是否都要同步。
