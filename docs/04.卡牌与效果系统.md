# 卡牌与效果系统

> 主题：Karen 卡牌开发的主入口文档  
> 适用范围：卡牌基类、命令、DynamicVar（动态变量）、Power（能力）、Clone、附魔与诅咒  
> 阅读建议：写新卡时先看本篇，再按需要查 `05` 和 `10`

## 卡牌基类

Karen 卡牌主要继承两条基类链：

- `KarenBaseCardModel : CardModel`
- `KarenDisableRelicBaseCardModel : KarenBaseCardModel`

### `KarenBaseCardModel` 提供了什么

除了原版 `CardModel` 就有的描述、数值、`OnPlay()`、升级逻辑之外，它额外暴露了这些系统钩子：

- `OnAddedToPromisePile()`
- `OnRemovedFromPromisePile()`
- `OnTurnEndInPromisePile()`
- `OnShineExhausted(PlayerChoiceContext, bool, CombatState)`
- `OnGlobalMove(PileType from, PileType to, AbstractModel? source)`
- `DoOption(PlayerChoiceContext, CardPlay)`

这意味着 Karen 卡牌可以在这些时机生效：

- 打出时
- 被放入约定牌堆时
- 从约定牌堆离开时
- 回合结束仍在约定牌堆中时
- 闪耀耗尽时
- 在任意牌堆间移动时
- 作为三选一选项牌被点击时

### `KarenDisableRelicBaseCardModel` 增加了什么

它在 `IsPlayable` 中加入了额外条件：

- 调用 `DisableRelicCmd.GetDisableableRelicCount(Owner)`
- 与自身的 `DisableRelicVar.IntValue` 比较
- 不满足条件则整张牌不可打出

也就是说，这类牌的“能不能打”不是单纯由能量、目标和原版限制决定，还会被禁用遗物数量约束。

## 一张牌最常见的实现结构

当前项目里，一张典型 Karen 卡通常会实现这些内容：

- 构造函数
  填能量、卡牌类型、稀有度、目标类型。
- `CanonicalTags`
  标注这张牌属于哪个机制。
- `CanonicalVars`
  提供描述文本和数值系统使用的动态变量。
- `OnPlay(...)`
  组织命令执行。
- `OnUpgrade()`
  定义升级成长。

代表性例子：

### `KarenShineStrike`

文件：`src/Core/Models/Cards/basic/KarenShineStrike.cs`

特点：

- 构造函数里直接 `this.AddShineMax(9)`，初始化闪耀值。
- `CanonicalTags` 带 `Strike`
- `CanonicalVars` 提供 `DamageVar(9m)`
- `OnPlay()` 用 `DamageCmd.Attack(...).Execute(choiceContext)` 组织攻击
- `OnUpgrade()` 直接把能量费用降 1

这个例子说明：

- Shine（闪耀）并不是通过单独组件挂上去，而是卡牌构造时直接初始化字段。
- 真正的闪耀显示和耗尽流程由系统补丁统一接管。

### `KarenPromiseDraw`

文件：`src/Core/Models/Cards/basic/KarenPromiseDraw.cs`

特点：

- 带 `KarenCustomEnum.PromisePileRelated`
- 先造成伤害，再调用 `PromisePileCmd.Draw(...)`
- 升级提升抽取数量

这个例子说明：

- 约定牌堆相关卡一般不直接操作内部卡堆，而是统一走 `PromisePileCmd`。

### `KarenMeetAgain`

文件：`src/Core/Models/Cards/promisePile/KarenMeetAgain.cs`

特点：

- 同时做格挡、抽牌、从手牌放入约定牌堆
- `OnPlay()` 里按顺序调用：
  - `CreatureCmd.GainBlock(...)`
  - `CardPileCmd.Draw(...)`
  - `PromisePileCmd.AddFromHand(...)`

这个例子体现了 Karen 卡牌的常见组织方式：

- 一张牌会串多个命令
- 每个命令分别负责一个子动作
- 逻辑顺序完全在 `OnPlay()` 中显式表达

## DynamicVar（动态变量）

项目里最常见的变量类型包括：

- `DamageVar`
- `BlockVar`
- `CardsVar`
- `PowerVar<T>`
- `EnergyVar`
- `DisableRelicVar`

它们主要承担三件事：

1. 给描述文本提供动态数值
2. 给升级差值显示提供基础
3. 给系统扩展暴露稳定的 UI 数值入口

例如 `DisableRelicVar`：

- 定义在 `src/Core/DisableRelicSystem/DisableRelicVar.cs`
- `VarName = "DisableRelic"`
- 本质就是一个自定义 `DynamicVar`

这类变量的优点是：

- 文本层和玩法层使用同一份数值源
- 不需要在描述文本里硬编码判断逻辑

## 常见标签与关键字

Karen 卡大量依赖 `CanonicalTags` 来声明自己属于哪些系统。

典型用途包括：

- PromisePile 相关牌：`KarenCustomEnum.PromisePileRelated`
- Shine 相关牌：`KarenCustomEnum.ShineRelated`
- DisableRelic 相关牌：`KarenCustomEnum.DisableRelicRelated`
- TmpStrength、Reward、Token 等机制标签

标签不仅影响卡牌分类，还影响：

- HoverTips
- 行内关键字显示
- 某些系统补丁是否对这张牌生效

例如 `CardHoverTipsPatch` 会根据卡牌标签和状态动态补出这些提示：

- `KAREN_SHINE`
- `KAREN_PROMISE_PILE`
- `KAREN_TMP_STRENGTH`
- `KAREN_DISABLE_RELIC`
- `KAREN_TOKEN_CARD`

## 命令与效果表达

Karen 卡牌效果仍以原生命令体系为主，再叠加项目自定义命令。

常见原生命令方向：

- 伤害：`DamageCmd`
- 格挡：`CreatureCmd.GainBlock`
- 抽牌 / 弃牌：`CardPileCmd`
- 附加 Power（能力）：`PowerCmd.Apply` 或相应 Creature 命令
- 自动打出 / 牌堆迁移：`CardCmd`、`CardPileCmd`

项目内常见自定义命令：

- `PromisePileCmd`
- `CardPileCmdEx`
- `CardSelectCmdEx`
- `DisableRelicCmd`
- `ExtraRewardCmd`
- `DrawPileIconCmd`

### 为什么推荐优先复用命令层

因为命令层已经帮你处理了很多系统兼容：

- 联机选择上下文
- Void / Infinite 等模式分支
- 历史记录和生成牌记录
- 视图与 Hook 接线

如果直接在 `OnPlay()` 里操作内部状态，往往容易绕过这些约定。

## PromisePile 相关写法

当前 PromisePile 卡最常见的命令入口有：

- `PromisePileCmd.Add(...)`
- `PromisePileCmd.Draw(...)`
- `PromisePileCmd.AddFromHand(...)`
- `PromisePileCmd.AddFromDiscard(...)`
- `PromisePileCmd.AddFromDraw(...)`
- `PromisePileCmd.SelectedToHand(...)`
- `PromisePileCmd.SwitchHand(...)`
- `PromisePileCmd.DiscardAll(...)`
- `PromisePileCmd.AutoPlayFromPromisePile(...)`

这些命令内部已经处理了特殊模式：

- Void 模式下，很多操作会被重定向到 `Draw` 牌堆
- InfiniteReinforcement 模式下，部分逻辑会改成直接抽牌或自动回填

所以写卡时不要自行复制这些模式判断，优先复用命令。

## Power（能力）

Karen Power 主要基于：

- `PowerModel`
- `KarenBasePower`

`KarenBasePower` 当前特别为 PromisePile 生命周期留了扩展点：

- `OnCardAddedToPromisePile`
- `OnCardRemovedFromPromisePile`
- `OnPromisePileEmpty`

代表性例子是 `KarenPromisePilePower`：

- 内部维护 `PromisePileMode` 的 Flags 组合
- 支持：
  - `Void`
  - `InfiniteReinforcement`
  - `Burn`
- 会动态生成 HoverTip，显示：
  - 当前模式说明
  - 当前约定牌堆中有哪些牌

如果一个效果本质上是持续状态、回合触发或 Hook（钩子）响应，优先考虑做成 Power（能力），而不是把状态塞进卡牌本体。

## Clone 与生成卡

项目内常见的复制与生成卡相关入口：

- `CreateClone`
- `CreateDupe`
- `RunState.CloneCard`
- `CardModelEx.CloneSafeForDeck`

需要特别注意：

- `SpireField` 附加字段不会天然等于“复制逻辑已经正确”
- 某些字段需要靠补丁手动同步

当前已经能看到的典型例子：

- Shine 系统给 `AbstractModel.MutableClone` 打了 `Postfix`
- 在克隆卡牌时手动把：
  - `ShineMax`
  - `ShineCurrent`
  复制到新实例

这说明：

- 只要你的新系统依赖 `SpireField`
- 且生成牌 / 克隆牌需要保留状态
- 你就要确认克隆链路是否已被显式补齐

## 作为选项牌的卡

Karen 基类提供了：

- `DoOption(PlayerChoiceContext, CardPlay)`

这说明部分 Karen 卡不是正常“打出”语义，而是被当作选择界面中的选项使用。

如果你要写这种牌：

- 不要只实现 `OnPlay()`
- 还要确认是否需要补 `DoOption()`

## 附魔与诅咒

除了卡牌与 Power（能力），本项目还会与这些系统交互：

- `Affliction`
- `Enchantment`

实际开发时，建议先判断一个新效果应该属于哪一层：

- 卡牌瞬时行为
- 持续能力
- 附魔 / 诅咒
- Karen 专属系统扩展

不要把所有状态都挤进一张牌的 `OnPlay()` 里。

## 开发检查表

新增一张卡时，通常至少要检查这些点：

1. 继承的基类是否正确。
2. `CanonicalVars` / `CanonicalKeywords` / `CanonicalTags` 是否齐全。
3. 文本是否和数值、升级逻辑一致。
4. 优先使用原生命令还是项目命令，是否选对。
5. 是否会与 Shine（闪耀）、PromisePile（约定牌堆）、GlobalMove（全局移牌）、DisableRelic（禁用遗物）联动。
6. 生成卡、复制卡和联机选择流程是否安全。
7. 需要的 HoverTip、标签和关键字是否已接线。
