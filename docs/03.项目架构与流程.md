# 项目架构与流程

> 主题：项目结构、模型关系与战斗主流程  
> 适用范围：理解代码目录、原生 STS2 架构和 KarenMod 的扩展切入点  
> 阅读建议：第一次接手项目时，优先在本篇和 `04`、`06` 之间来回对照

---

## 项目入口

项目的接入点主要有四个：

- `MainFile.cs`
  Mod 初始化与 `Harmony.PatchAll()` 入口。
- `ModelDbAllCharactersPatch`
  把 Karen 注入角色列表，并清理相关缓存。
- `src/Core/Models/Characters/Karen.cs`
  角色定义。
- `src/Core/Models/CardPools/KarenCardPool.cs`
  卡池聚合入口。

其中真正把功能“铺开”的不是角色类本身，而是各系统目录和 `src/Core/Patches/` 中的大量补丁。

## 代码分层

当前代码可以粗分成五层：

### 1. 模型层

位于 `src/Core/Models/`，负责定义游戏对象本身：

- `CardModel`
- `PowerModel`
- `CharacterModel`
- `RelicModel`
- 各类卡池 / 遗物池

Karen 侧常见基类有：

- `KarenBaseCardModel : CardModel`
- `KarenDisableRelicBaseCardModel : KarenBaseCardModel`
- `KarenBasePower : PowerModel`

### 2. 命令层

位于 `src/Core/Commands/`，负责组织可复用的行为入口。

当前自定义命令文件至少包括：

- `PromisePileCmd`
- `CardPileCmdEx`
- `CardSelectCmdEx`
- `DisableRelicCmd`
- `DrawPileIconCmd`
- `ExtraRewardCmd`

这层的作用是把“卡牌想做什么”从“底层如何接线”里分离出来。

### 3. 系统层

位于这些目录：

- `src/Core/ShineSystem/`
- `src/Core/PromisePileSystem/`
- `src/Core/GlobalMoveSystem/`
- `src/Core/ExtraReplaySystem/`
- `src/Core/DisableRelicSystem/`
- `src/Core/SaveSystem/`

这层负责做原版没有的额外规则和运行时状态管理。

### 4. 补丁层

位于 `src/Core/Patches/` 及各系统子目录的 `Patches/` 下。

补丁层负责：

- 让系统层逻辑真正接入原版流程
- 改写原版结果牌堆或播放次数
- 修正视图 / 文本 / 交互层问题

### 5. 资源与表现层

位于：

- `scenes/`
- `images/`
- `materials/`
- `audio/`
- `localization/`

这里和 Godot 节点、界面、音效、本地化文本直接对应。

## STS2 原生架构如何被利用

KarenMod 主要依托以下原生概念：

- `AbstractModel`
  所有模型对象的共同基类。
- `CardModel`
  卡牌核心入口，`OnPlay`、描述、标签、关键字等都从这里延展。
- `PowerModel`
  持续状态、回合钩子和数值层常用入口。
- `Hook`
  原生事件钩子总线。
- `ModelDb`
  全局模型注册与检索中心。

KarenMod 的总体策略不是绕开原版，而是在原版流程上的关键节点做局部扩展：

- 用继承扩展模型对象
- 用 `SpireField` / `SavedSpireField` 存储附加状态
- 用 Harmony 补丁插入或重定向流程

## 关键模型关系

### 卡牌

- `KarenBaseCardModel`
  是 Karen 卡牌的核心公共基类。
- 它不仅继承自 `CardModel`，还额外暴露了多个系统扩展钩子：
  - `OnAddedToPromisePile()`
  - `OnRemovedFromPromisePile()`
  - `OnTurnEndInPromisePile()`
  - `OnShineExhausted(...)`
  - `OnGlobalMove(...)`
  - `DoOption(...)`

这意味着 Karen 卡牌的自定义行为不只发生在 `OnPlay()`，还可能在牌堆移动、闪耀耗尽、约定牌堆生命周期里触发。

### Power

- `KarenBasePower`
  是 Karen Power 的公共扩展基类。
- 当前它主要为 PromisePile 生命周期提供扩展钩子：
  - `OnCardAddedToPromisePile`
  - `OnCardRemovedFromPromisePile`
  - `OnPromisePileEmpty`

### 角色

- `Karen : CharacterModel`
  负责定义起始卡组、起始遗物、颜色、动画延迟和部分表现层配置。

## 运行流程

### 初始化流程

```text
游戏加载 Mod
 -> MainFile.Initialize()
 -> 注册配置
 -> Harmony.PatchAll()
 -> ModelDb.AllCharacters 被访问
 -> ModelDbAllCharactersPatch 注入 Karen
```

### 打牌主流程

```text
玩家打牌
 -> CardModel.OnPlayWrapper
 -> Hook.BeforeCardPlayed
 -> CardModel.OnPlay
 -> Power / 遗物 / 附魔 / 诅咒响应
 -> Hook.AfterCardPlayed
 -> 结果牌堆结算
 -> Hook.AfterCardChangedPiles
```

KarenMod 主要在这些点插入自定义逻辑：

- `CardModel.OnPlayWrapper`
  Shine（闪耀）值递减。
- `Hook.ModifyCardPlayCount`
  ExtraReplay（额外重播）修改实际播放次数。
- `Hook.ModifyCardPlayResultPileTypeAndPosition`
  当卡牌应进入 Shine 耗尽流程时，重定向其结果牌堆。
- `Hook.AfterCardChangedPiles`
  GlobalMove（全局移牌）监听与约定牌堆追踪。

## 牌堆移动语义

项目里“牌堆移动”是一个很核心的扩展点。

`GlobalMovePatch` 当前的处理方式是：

- 普通牌堆移动直接读取 `oldPile` 和当前 `card.Pile?.Type`
- 如果新牌堆是 `Play`，先缓存“打出前来源牌堆”
- 如果后续再从 `Play` 流向别处，则把两段移动拼成一次完整语义

例如：

- `Hand -> Play`
- `Play -> Discard`

最终会被合并理解为：

- `Hand -> Discard`

这能让卡牌的 `OnGlobalMove(...)` 收到更稳定、更接近玩法语义的事件，而不是被播放区中转噪音干扰。

## 存档流程

Karen 的自定义数据不是单独写伴随文件，而是嵌入原版存档 JSON。

从 `RunSaveManager_Patches.cs` 和 `KarenModSaveBuffer.cs` 可以看出当前流程是：

```text
保存
 -> 收集 Shine 数据
 -> 收集 ShinePile 数据
 -> 写入 karen_mod_data 字段

读取
 -> 从 JSON 里提取 karen_mod_data
 -> 暂存到 KarenModSaveBuffer
 -> RunManager.SetUpSavedSinglePlayer / MultiPlayer 完成后再消费
 -> 恢复到当前 RunState / Deck 对应的 CardModel 实例
```

这个设计的好处是：

- 不需要自定义伴随存档文件
- 恢复时机和原版一致
- 单机和联机都能共用一套结构

## 架构上的几个特点

### 1. 系统状态大量依赖 `SpireField`

例如：

- Shine 当前值 / 最大值
- 是否应进入 ShinePile
- PromisePile 附着在玩家战斗状态上的卡堆
- ExtraReplay 的额外播放次数

这让扩展能附着在原版对象上，而不需要篡改原版类定义。

### 2. 命令层比直接写 `OnPlay` 更重要

当前很多复杂玩法都不是把所有逻辑直接塞进卡牌 `OnPlay`，而是通过：

- `PromisePileCmd`
- `CardSelectCmdEx`
- `ExtraRewardCmd`

这些命令层来复用。

### 3. 视图层和规则层是分开的

例如 PromisePile：

- 规则逻辑在 `PromisePileManager`
- 牌堆显示和 VFX 在 PromisePile 相关 VFX / Screen / Patch

例如 Karen 能量计数器：

- 自定义节点 `SNEnergyCounter`
- 原版 `NEnergyCounter` 的兼容补丁负责接线
