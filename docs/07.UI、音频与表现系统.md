## 表现层结构

Karen 的表现层不是单个目录能说完的，而是分散在：

- `src/Core/Patches/`
- `src/Core/Audio/`
- `src/Core/Node/`
- `scenes/`
- `images/`
- `materials/`
- `localization/`

当前代码说明 Karen 的表现实现思路是：

- 能复用原版界面的地方，优先用补丁接线
- 需要自定义节点的地方，再挂自己的 Godot 节点
- 音效走单独的 `KarenAudioManager`

## HoverTip 与关键字提示

相关文件：

- `src/Core/Patches/CardHoverTipsPatch.cs`
- `src/Core/Patches/CardKeywordInlinePatch.cs`

### `CardHoverTipsPatch`

这个补丁给 `CardModel.HoverTips` 做了后处理，根据卡牌标签和状态动态追加提示。

当前已经接了这些关键词：

- `KAREN_SHINE`
- `KAREN_PROMISE_PILE`
- `KAREN_SHINE_CARD_REWARD`
- `KAREN_TMP_STRENGTH`
- `KAREN_RETAIN_TMP_STRENGTH`
- `KAREN_DISABLE_RELIC`
- `KAREN_TOKEN_CARD`

它的特点是：

- 不要求卡牌手工一张张写 HoverTip 列表
- 只要标签和状态接对，提示就能自动出现

### `CardKeywordInlinePatch`

这个补丁处理的是描述文本行内显示问题。

当前已处理的一个具体场景是：

- 当 `Innate` 和 `Ethereal` 同时出现在同一张 Karen 卡上时
- 去掉 badge 之间的换行
- 让它们在同一行显示

这说明项目已经开始关注“文本排版层”的细节，而不只是规则正确。

## Karen 能量计数器

相关文件：

- `src/Core/Node/SNEnergyCounter.cs`
- `src/Core/Patches/NEnergyCounterReadyPatch.cs`
- `src/Core/Patches/NEnergyCounterOnEnergyChangedPatch.cs`
- `scenes/combat/energy_counters/karen_energy_counter.tscn`

### 自定义节点

`SNEnergyCounter` 继承自原版 `NEnergyCounter`，在 `_Ready()` 中会：

- 先调用 `base._Ready()`
- 用 `AccessTools.Field(...)` 把原版 `_backVfx` / `_frontVfx` 置空
- 读取自己的：
  - `%MyEnergyVfxBack`
  - `%MyEnergyVfxFront`

这说明 Karen 的能量特效不是简单覆盖原节点，而是显式替换原版特效引用。

### 兼容补丁

`NEnergyCounterReadyPatch` 通过 `Transpiler` 把 `_Ready()` 中的：

- `GetNode<T>()`

替换成：

- `GetNodeOrNull<T>()`

作用是避免 Karen 自定义能量场景缺少某些原版节点时直接报错。

`NEnergyCounterOnEnergyChangedPatch` 则在能量增加时：

- 判断实例是不是 `SNEnergyCounter`
- 如果是，则重启 `_myBackVfx` / `_myFrontVfx`

说明 Karen 自定义能量特效在“数值变化”这一层也接上了原版时机。

## 音频系统

相关文件：

- `src/Core/Audio/KarenAudioManager.cs`
- `src/Core/Audio/KarenFormMusicManager.cs`
- `src/Core/Patches/KarenFormMusicRoomPatch.cs`

### 当前能力

`KarenAudioManager` 当前支持：

- `Play(...)`
  播放一次性短音效，结束后自动释放 `AudioStreamPlayer`
- `PlayLoop(...)`
  播放循环音效，需要外部手动停止和释放
- `GetDuration(...)`
  获取音频时长
- `ClearCache()`
  清理缓存

### 设计特点

- 资源路径走 `res://ShoujoKagekiAijoKaren/audio/...`
- 内部缓存 `AudioStream`
- 统一挂到 Godot `SceneTree.Root`
- 总线固定使用 `SFX`

这样做的好处是：

- 音量自动跟随游戏 SFX 设置
- 不依赖 Godot 资源导入系统也能直接用字节加载

支持的格式包括：

- `.ogg`
- `.mp3`
- `.wav`

## STS2 本体 BGM 控制器注意事项

本节记录一次 Karen Form BGM 与本体房间音乐冲突排查的结论。

相关本体代码路径：

- `D:\Github\spire-codex\extraction\raw\src\Core\Nodes\Audio\NRunMusicController.cs`
- `D:\Github\spire-codex\extraction\raw\src\gdscript\music_controller_proxy.gd`
- `D:\Github\spire-codex\extraction\raw\src\Core\Runs\RunManager.cs`

参考库：

- `D:\Github\STS2-RitsuLib\Audio\AudioVanillaBridge.cs`
- `D:\Github\BaseLib-StS2-master\docs\FmodAudio.md`

### 崩溃现象

进入 RestSite 时可能出现原生崩溃：

```text
Fatal error. 0xC0000005
at Godot.GodotObject.Call(...)
at MegaCrit.Sts2.Core.Nodes.Audio.NRunMusicController.UpdateTrack()
at MegaCrit.Sts2.Core.Runs.RunManager.EnterRoomInternal(...)
```

这不是普通 C# 异常，而是 Godot / FMOD 原生层访问冲突。`try/catch` 通常抓不到。

本体 `RunManager.EnterRoomInternal` 在房间进入后会调用：

```csharp
NRunMusicController.Instance?.UpdateTrack();
```

而 `UpdateTrack()` 在 RestSite 时会执行：

```csharp
UpdateTrack("Progress", (float)MusicProgressTrack.Rest);
_proxy.Call("update_campfire_ambience", 0);
```

如果音乐 event、ambience event 或 act bank 状态已经被 mod 破坏，这里可能在 Godot / FMOD 原生层崩溃。

### 不要直接调用 `NRunMusicController.StopMusic()`

本体 C# 的 `NRunMusicController.StopMusic()` 不只是停止音乐。它会：

```csharp
_proxy.Call(_stopMusic);
_proxy.Call(_stopAmbience);
UnloadActBanks();
```

其中 `UnloadActBanks()` 会卸载当前 act 的音乐 bank。之后本体继续 `UpdateTrack()` 或 RestSite 的 `update_campfire_ambience` 时，底层 FMOD 状态可能不完整。

如果 mod 只是想临时压掉本体 BGM，不要调用：

```csharp
NRunMusicController.Instance?.StopMusic();
```

### 只停当前音乐 event 的方式

本体 GDScript `music_controller_proxy.gd` 里的 `stop_music()` 只会停当前音乐 event：

```gdscript
func stop_music():
    if _musicEv != null:
        _musicEv.stop(0)
        _musicEv.release()
        _musicEv = null
```

它不会卸载 act banks，也不会停止 ambience。

当前 Karen Form BGM 采用的策略是：

```csharp
var controller = NRunMusicController.Instance;
var proxy = Traverse.Create(controller).Field<Node>("_proxy").Value;
proxy.Call("stop_music");
```

这样可以临时停止本体 BGM，让 Karen 自己的音乐播放，同时保留 act bank 和 ambience 状态。

### 停掉本体音乐后必须主动恢复

本体进入新房间时通常只调用 `UpdateTrack()`，不会自动重建 `_musicEv`。如果 mod 之前调用过 `proxy.stop_music()`，新房间只更新参数不会让音乐重新响起。

RitsuLib 的恢复方案是：

```csharp
controller.UpdateMusic();
controller.UpdateTrack();
controller.UpdateAmbience();
```

这对应 `D:\Github\STS2-RitsuLib\Audio\AudioVanillaBridge.cs` 的 `RefreshRunMusic()`。

Karen 当前在 `KarenFormMusicManager` 中维护 `_needsGameMusicRestore` 标记：

- Form BGM 开始时，只停本体 music event。
- Form BGM 正常结束时，延迟一帧恢复本体音乐。
- `StopForCutscene()` 只停音乐并标记待恢复，避免演出期间 BGM 立刻回来。
- `KarenFormMusicRoomPatch` 在 `Hook.AfterRoomEntered` 后兜底调用恢复，解决“进入新房间没有自动播放音乐”的情况。

### 经验规则

处理 STS2 BGM 时优先遵守：

1. 不要把 `StopMusic()` 当作“暂停 BGM”。它会卸载 banks。
2. 想临时停 BGM，调用底层 proxy 的 `stop_music()` 更安全。
3. 停掉 `_musicEv` 后，后续必须用 `UpdateMusic()` 重建事件。
4. 恢复时按 RitsuLib 顺序执行：`UpdateMusic()`、`UpdateTrack()`、`UpdateAmbience()`。
5. RestSite 特别敏感，因为它会额外调用 `update_campfire_ambience`。
6. 原生崩溃堆栈如果停在 `GodotObject.Call`，要优先怀疑 GDScript proxy / FMOD bank / event 状态，而不是普通 C# 空引用。

## PromisePile / Shine 的表现层联动

虽然规则在系统层，但 PromisePile 和 Shine 都已经影响到表现层：

- PromisePile 会更新星星数量显示
- PromisePile 能打开专门的牌堆查看界面
- Void 模式会覆盖抽牌堆图标
- Shine 耗尽时会播放专门的耗尽动画

这意味着改系统时不能只看规则代码，很多时候还要反查表现接线是否同步更新。

## 排查表现问题时的顺序

如果界面或特效异常，建议按这个顺序排查：

1. 规则层状态有没有真的更新。
2. 是否已有对应 `Cmd` 或 Manager 更新视图。
3. 节点是原版节点被补丁接线，还是自定义 Godot 节点。
4. 是否还需要同步处理 HoverTip、关键字、音效和图标。
