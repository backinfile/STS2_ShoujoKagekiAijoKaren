# 工程技术与BaseLib

> 主题：工程层实现手法与常用技术参考  
> 适用范围：Harmony、异步补丁、反射、BaseLib 扩展点  
> 阅读建议：做补丁或底层调试时，把本篇和 `03`、`06` 对照来看

---

## Harmony 补丁

本项目大量依赖 Harmony 补丁接入原版流程。

当前实际能看到的补丁类型包括：

- Prefix
- Postfix
- ReversePatch
- Transpiler

### 当前常见用途

- 在 `OnPlayWrapper` 前后插入 Karen 规则
- 改写结果牌堆
- 改写打牌次数
- 修正视图节点获取方式
- 追加 HoverTip 或描述文本
- 截获存档读写流程

### 代表性补丁

- `ShinePatch`
  改写闪耀值递减和结果牌堆逻辑。
- `ExtraReplayPatch`
  改写 `Hook.ModifyCardPlayCount`。
- `GlobalMovePatch`
  监听 `Hook.AfterCardChangedPiles`。
- `NEnergyCounterReadyPatch`
  用 `Transpiler` 把 `GetNode` 改成 `GetNodeOrNull`。
- `CardHoverTipsPatch`
  给 `CardModel.HoverTips` 动态追加内容。
- `RunSaveManager_Patches`
  截获保存和读取流程，注入 / 取出 `karen_mod_data`。

## Async 补丁

项目里有不少异步链路，尤其涉及：

- 选牌
- 牌堆迁移
- 自动打出
- 自定义系统触发

当前常见做法包括：

- `Async.Postfix(...)`
- `Async.Prefix(...)`

从现有代码看，使用原则是：

- 不随意开独立线程
- 尽量在原版异步流程上继续串接
- 保持 Godot 主线程和原版状态机上下文不被破坏

例如 `GlobalMovePatch` 中，就使用 `Async.Postfix(ref __result, ...)` 把自定义触发挂回原始异步链。

## ReversePatch 的用途

`ShinePatch` 当前包含一个典型 `HarmonyReversePatch` 用法：

- 给 `CardPileCmd.Add(...)` 的原方法留一个 `OriginalMethodStub(...)`
- 在接管部分卡牌迁移逻辑后，仍然能对剩余卡牌调用“原版实现”

这种写法适合：

- 你只想接管一部分输入
- 但不想整段复制原版方法逻辑

## Transpiler 的用途

当前最典型的 `Transpiler` 是：

- `NEnergyCounterReadyPatch`

它不改业务逻辑，只做一个小而稳的指令替换：

- 把 `GetNode<T>()` 改成 `GetNodeOrNull<T>()`

这类补丁适合：

- 原逻辑整体正确
- 只是某个 API 调用太脆弱

## AccessTools 与反射

当前项目大量使用 `AccessTools` 访问原版私有成员。

典型场景包括：

- 在 `MainFile` 里清 `ModelDb` 私有缓存字段
- 在 `SNEnergyCounter` 里操作 `NEnergyCounter` 的私有 VFX 字段
- 在补丁里定位私有方法或嵌套类型

适合使用反射的情况：

- 原版没有公开入口
- 你只是需要很小范围地接线

不适合的情况：

- 逻辑本身可以通过现有公开 API 实现

## BaseLib

项目内最常见的 BaseLib 用法有：

- `SpireField`
- `SavedSpireField`
- `CustomEnum`

### `SpireField`

主要用于给原版对象动态挂状态，例如：

- 卡牌的 Shine 当前值 / 最大值
- 卡牌的额外重播次数
- PromisePile 挂到玩家战斗状态上
- GlobalMove 的打牌中转缓存

### `SavedSpireField`

本轮代码里最重要的结论是：

- 不是所有状态都直接靠 `SavedSpireField` 自动持久化
- Karen 这里很多关键状态是通过自定义存档 JSON 手工保存 / 恢复的

换句话说，不要误以为“用了 SpireField 就天然有存档”。

### `CustomEnum`

主要用于补充项目自定义枚举值，例如特殊牌堆类型。

PromisePile 和 Shine 相关流程里都有这类扩展枚举参与。

## BaseLib Mod 配置按钮参考

`D:\Github\BaseLib-StS2-master` 里的 Mod 设置界面按钮不是旧版 StS1 的 `BaseMod.registerModBadge` 接法，而是通过 Harmony 直接接入 STS2 的 Godot UI。

### 设置入口按钮

入口补丁位于 BaseLib 的 `Patches/Utils/ModConfigPatch.cs`。

核心做法有两种：

- 主菜单入口：
  `InjectMainMenuModConfigPatch` 在 `NMainMenu._Ready` 时复制原版 `SettingsButton`，改名为 `ModConfigButton`，绑定 `NClickableControl.SignalName.Released`，点击后调用 `mainMenu.SubmenuStack.PushSubmenuType<NModConfigSubmenu>()`。
- 设置页入口：
  `InjectSettingsModConfigPatch` 在 `NSettingsScreen._Ready` 时复制原版 `Modding` 行，改成 `BaseLibModConfig`，把按钮文案改成 `Open Config`，点击后同样 push 到 `NModConfigSubmenu`。

因此入口层的关键点是：

- 复用原版已有按钮节点，避免从零搭 UI 风格。
- 用 `Connect(NClickableControl.SignalName.Released, Callable.From<...>(...))` 绑定点击回调。
- 在回调里通过 `NMainMenuSubmenuStack.PushSubmenuType<NModConfigSubmenu>()` 打开 BaseLib 自己的配置子菜单。

### 配置子菜单

子菜单类是 `Config/UI/NModConfigSubmenu.cs`。

它负责：

- 左侧列出所有已注册且有可见设置项的 `ModConfig`。
- 用 `NModListButton` 表示左侧 mod 列表按钮。
- 点击某个 mod 后调用 `LoadModConfig(modConfig)`。
- `LoadModConfig` 内部创建右侧容器，然后调用 `config.SetupConfigUI(_optionContainer)` 让对应 mod 填充自己的设置项。

配置注册入口是 `Config/ModConfigRegistry.cs`：

- `ModConfigRegistry.Register(modId, config)` 会保存配置对象。
- `GetAll()` 会按 `ModId` 排序返回所有配置。
- `NModConfigSubmenu` 只显示 `HasVisibleSettings()` 为 true 的配置。

### 配置页内的按钮控件

具体按钮控件是 `Config/UI/NConfigButton.cs`。

它继承 `NSettingsButton`，构造时手动创建背景图、文字 `Label` 和选择框。真正的行为在 `Initialize` 里绑定：

```csharp
public void Initialize(string buttonText, Action onPressed)
{
    SetColor(Color.FromHtml(DefaultColor));
    _onPressedAction = onPressed;

    var label = GetNodeOrNull<Label>("Label");
    if (label != null)
    {
        label.Text = buttonText;
    }

    Connect(NClickableControl.SignalName.Released, Callable.From<NConfigButton>(OnReleased));
}

private void OnReleased(NConfigButton button)
{
    _onPressedAction?.Invoke();
}
```

也就是说，配置按钮的本质是：

- 显示文本由 `Initialize(buttonText, ...)` 写入。
- 点击事件走 `NClickableControl.SignalName.Released`。
- 实际逻辑由 `Action onPressed` 注入。
- `_ExitTree` 里会断开信号，避免节点销毁后残留连接。

### 给 Mod 配置页添加按钮

通常不需要直接 new `NConfigButton`，而是继承 `SimpleModConfig` 后使用它的封装。

`Config/SimpleModConfig.cs` 支持两种方式：

- 手动调用 `CreateButton(rowLabelKey, buttonLabelKey, onPressed, addHoverTip)`。
- 给方法加 `[ConfigButton("ButtonLabelKey")]`，让 `GenerateButtonRowFromMethod` 自动生成按钮行。

BaseLib 自己的例子在 `Config/BaseLibConfig.cs`：

```csharp
[ConfigButton("HarmonyDumpNow")]
public static void HarmonyDumpWriteNow(ModConfig _)
{
    HarmonyPatchDumpCoordinator.TryManualDumpFromSettings();
}
```

自动生成链路是：

- `SimpleModConfig.SetupConfigUI(...)`
- `GenerateOptionsForAllProperties(...)`
- 扫描带 `[ConfigButton]` 的方法
- `GenerateButtonRowFromMethod(...)`
- `CreateButton(...)`
- `CreateRawButtonControl(...)`
- `NConfigButton.Initialize(...)`

如果后续本项目要加 Mod 设置按钮，优先参考这条链路：用 `SimpleModConfig` 暴露配置项和 `[ConfigButton]` 方法，再让 BaseLib 的配置 UI 自动生成按钮；只有需要完全自定义控件外观时，再直接参考 `NConfigButton`。

### 按钮执行后的 Modal Popup 提示

BaseLib 源码里可以看到两类可用的弹窗方案：

- `NGenericPopup`
  BaseLib 在 `SimpleModConfig.ConfirmRestoreDefaults()` 中用它做“恢复默认设置”的确认框。
- `NErrorPopup`
  BaseLib 在 `ModConfig.ShowAndClearPendingErrors()` 中用它显示配置错误，消息来源是 `ModConfig.ModConfigLogger.PendingUserMessages`。

如果只是提示玩家“生成完成 / 生成失败”，不要优先用 `ModConfigLogger.Error/Warn(..., showInGui: true)`，否则弹窗标题和语义会偏向错误提示。

当前项目里已经采用兼容性更稳的 Godot 原生 `AcceptDialog`：

```csharp
private static void ShowMessage(string title, string message)
{
    if (NGame.Instance == null)
    {
        MainFile.Logger.Info($"{title}: {message}");
        return;
    }

    var dialog = new AcceptDialog
    {
        Title = title,
        DialogText = message,
        MinSize = new Vector2I(560, 220),
    };

    dialog.Confirmed += dialog.QueueFree;
    dialog.Canceled += dialog.QueueFree;
    dialog.CloseRequested += dialog.QueueFree;

    NGame.Instance.AddChild(dialog);
    dialog.PopupCentered();
}
```

这套写法适合：

- 配置按钮执行完成后提示结果。
- 失败时显示简短错误，并把详细信息写日志。
- 当前 BaseLib NuGet 版本还没有新版 `ConfigButton` / popup helper API 时保持兼容。

需要注意：

- 弹窗节点要挂到 `NGame.Instance` 或当前有效 UI 根节点下。
- 用完后通过 `QueueFree` 释放，避免重复点击后残留节点。
- 长耗时按钮建议执行期间禁用按钮，完成后再恢复。

## 存档工程实现

当前 `RunSaveManager_Patches.cs` 已经把 Karen 数据嵌入到原版 `.save` 文件里。

可以概括为：

- 写入时：
  - 收集 Shine 数据
  - 收集 ShinePile 数据
  - 注入 `karen_mod_data`
- 读取时：
  - 从 JSON 里提取 `karen_mod_data`
  - 缓存到 `KarenModSaveBuffer`
  - 等 `RunManager.SetUpSavedSinglePlayer / MultiPlayer` 完成后再恢复

这是当前项目工程层最成熟的一块扩展之一，建议新增可持久化系统时优先参考它。
