## Harmony 补丁

本项目大量依赖 Harmony 补丁接入原版流程。

当前实际能看到的补丁类型包括：

- Prefix
- Postfix
- ReversePatch
- Transpiler

### 当前常见用途

- 在 `OnPlayWrapper` 前后插入 Karen 规则
- 改写结果牌堆
- 改写打牌次数
- 修正视图节点获取方式
- 追加 HoverTip 或描述文本
- 截获存档读写流程

### 代表性补丁

- `ShinePatch`
  改写闪耀值递减和结果牌堆逻辑。
- `ExtraReplayPatch`
  改写 `Hook.ModifyCardPlayCount`。
- `GlobalMovePatch`
  监听 `Hook.AfterCardChangedPiles`。
- `NEnergyCounterReadyPatch`
  用 `Transpiler` 把 `GetNode` 改成 `GetNodeOrNull`。
- `CardHoverTipsPatch`
  给 `CardModel.HoverTips` 动态追加内容。
- `RunSaveManager_Patches`
  截获保存和读取流程，注入 / 取出 `karen_mod_data`。

## Async 补丁

项目里有不少异步链路，尤其涉及：

- 选牌
- 牌堆迁移
- 自动打出
- 自定义系统触发

当前常见做法包括：

- `Async.Postfix(...)`
- `Async.Prefix(...)`

从现有代码看，使用原则是：

- 不随意开独立线程
- 尽量在原版异步流程上继续串接
- 保持 Godot 主线程和原版状态机上下文不被破坏

例如 `GlobalMovePatch` 中，就使用 `Async.Postfix(ref __result, ...)` 把自定义触发挂回原始异步链。

## ReversePatch 的用途

`ShinePatch` 当前包含一个典型 `HarmonyReversePatch` 用法：

- 给 `CardPileCmd.Add(...)` 的原方法留一个 `OriginalMethodStub(...)`
- 在接管部分卡牌迁移逻辑后，仍然能对剩余卡牌调用“原版实现”

这种写法适合：

- 你只想接管一部分输入
- 但不想整段复制原版方法逻辑

## Transpiler 的用途

当前最典型的 `Transpiler` 是：

- `NEnergyCounterReadyPatch`

它不改业务逻辑，只做一个小而稳的指令替换：

- 把 `GetNode<T>()` 改成 `GetNodeOrNull<T>()`

这类补丁适合：

- 原逻辑整体正确
- 只是某个 API 调用太脆弱

## AccessTools 与反射

当前项目大量使用 `AccessTools` 访问原版私有成员。

典型场景包括：

- 在 `MainFile` 里清 `ModelDb` 私有缓存字段
- 在 `SNEnergyCounter` 里操作 `NEnergyCounter` 的私有 VFX 字段
- 在补丁里定位私有方法或嵌套类型

适合使用反射的情况：

- 原版没有公开入口
- 你只是需要很小范围地接线

不适合的情况：

- 逻辑本身可以通过现有公开 API 实现

## BaseLib

项目内最常见的 BaseLib 用法有：

- `SpireField`
- `SavedSpireField`
- `CustomEnum`

### `SpireField`

主要用于给原版对象动态挂状态，例如：

- 卡牌的 Shine 当前值 / 最大值
- 卡牌的额外重播次数
- PromisePile 挂到玩家战斗状态上
- GlobalMove 的打牌中转缓存

### `SavedSpireField`

本轮代码里最重要的结论是：

- 不是所有状态都直接靠 `SavedSpireField` 自动持久化
- Karen 这里很多关键状态是通过自定义存档 JSON 手工保存 / 恢复的

换句话说，不要误以为“用了 SpireField 就天然有存档”。

### `CustomEnum`

主要用于补充项目自定义枚举值，例如特殊牌堆类型。

PromisePile 和 Shine 相关流程里都有这类扩展枚举参与。

## 存档工程实现

当前 `RunSaveManager_Patches.cs` 已经把 Karen 数据嵌入到原版 `.save` 文件里。

可以概括为：

- 写入时：
  - 收集 Shine 数据
  - 收集 ShinePile 数据
  - 注入 `karen_mod_data`
- 读取时：
  - 从 JSON 里提取 `karen_mod_data`
  - 缓存到 `KarenModSaveBuffer`
  - 等 `RunManager.SetUpSavedSinglePlayer / MultiPlayer` 完成后再恢复

这是当前项目工程层最成熟的一块扩展之一，建议新增可持久化系统时优先参考它。
