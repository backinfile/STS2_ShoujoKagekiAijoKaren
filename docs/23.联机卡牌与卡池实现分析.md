## 1. 文档目的

这份文档整理两部分内容：

1. `docs/基本信息.md` 中和当前开发最相关的基础信息。
2. 游戏本体中“联机卡牌效果是怎么写的”以及“联机卡池是怎么处理的”。

本文重点不是复述联机网络架构全貌，而是说明：

- 联机卡牌在源码里如何定义

- 联机模式下卡池、奖励牌、商店牌、生成牌如何过滤

## 3. 游戏本体里联机卡牌是怎么写效果的

### 3.1 结论

游戏本体并没有单独做一套“联机卡牌脚本系统”。

联机卡牌仍然是普通 `CardModel` 子类，效果仍然写在卡牌自己的 `OnPlay`、Power、Hook 里。联机和单机的主要区别不在“效果写法”，而在：

- 卡牌是否带 `MultiplayerConstraint`
- 打牌动作是否通过联机同步器排队执行
- 效果里是否会访问“队友/其他玩家”的对象

### 3.2 卡牌效果的标准入口

本体卡牌基类：

- `D:\Github\spire-codex\extraction\raw\src\Core\Models\CardModel.cs`

核心点：

- 卡牌效果主入口是 `protected virtual Task OnPlay(PlayerChoiceContext choiceContext, CardPlay cardPlay)`
- 各张牌通过 `override OnPlay(...)` 实现自身效果
- 目标、费用、关键词、动态变量、打出后去向等，也都仍然由 `CardModel` 统一管理

也就是说，联机牌和平常牌在“写法”上没有分家，依旧是：

1. 定义卡牌类
2. 配置 `CardType / CardRarity / TargetType / DynamicVar / Keywords`
3. 在 `OnPlay(...)` 中调用命令系统或 Power 系统

### 3.3 联机下打牌为什么还能同步

联机同步不要求卡牌自己发送网络消息。

在 `CardModel.EnqueueManualPlay(...)` 中，真正执行的是：

```csharp
RunManager.Instance.ActionQueueSynchronizer.RequestEnqueue(new PlayCardAction(this, target));
```

对应同步器源码：

- `D:\Github\spire-codex\extraction\raw\src\Core\GameActions\Multiplayer\ActionQueueSynchronizer.cs`

它的机制是：

- Client 不直接本地决定动作顺序
- Client 先向 Host 请求入队
- Host 负责确认顺序并广播
- 所有端再按同样顺序执行同一个 `GameAction`

因此，卡牌效果代码本身可以保持普通写法，因为网络层同步的是“动作请求与执行顺序”，而不是要求每张卡自己实现网络协议。

### 3.4 联机卡牌的识别方式

联机约束枚举定义在：

- `D:\Github\spire-codex\extraction\raw\src\Core\Entities\Cards\CardMultiplayerConstraint.cs`

定义如下：

- `None`
- `MultiplayerOnly`
- `SingleplayerOnly`

在卡牌类里通过覆写属性声明：

```csharp
public override CardMultiplayerConstraint MultiplayerConstraint => CardMultiplayerConstraint.MultiplayerOnly;
```

这表示该牌只应在多人模式下出现。

### 3.5 联机牌的常见写法模式

联机牌本质上仍是普通卡牌，只是效果更常涉及队友、共享战斗状态或玩家间联动。

常见模式有三类。

#### 3.5.1 直接指定队友为目标

示例：

- `D:\Github\spire-codex\extraction\raw\src\Core\Models\Cards\Coordinate.cs`

特点：

- `TargetType.AnyAlly`
- 在 `OnPlay(...)` 中直接对队友施加 Power

核心代码思路：

```csharp
await PowerCmd.Apply<CoordinatePower>(cardPlay.Target, baseValue, base.Owner.Creature, this);
```

这类牌本质上和普通增益牌一样，只不过目标不是自己或敌人，而是联机队友。

#### 3.5.2 遍历所有队友执行效果

示例：

- `D:\Github\spire-codex\extraction\raw\src\Core\Models\Cards\HuddleUp.cs`

特点：

- 通过 `CombatState.GetTeammatesOf(...)` 找到当前角色的队友
- 对所有存活的玩家队友逐个生效

核心思路：

```csharp
foreach (Creature item in teammates)
{
    await CardPileCmd.Draw(choiceContext, amount, item.Player);
}
```

这说明联机牌并不依赖特殊“联机卡效果接口”，而是直接在正常效果逻辑里遍历其他玩家对象。

#### 3.5.3 卡牌本体挂 Power，再由 Power 做联动

示例：

- 卡：`D:\Github\spire-codex\extraction\raw\src\Core\Models\Cards\TagTeam.cs`
- Power：`D:\Github\spire-codex\extraction\raw\src\Core\Models\Powers\TagTeamPower.cs`

实现方式：

1. 卡牌先正常造成伤害
2. 再给目标挂一个 `TagTeamPower`
3. `TagTeamPower.ModifyCardPlayCount(...)` 负责在后续队友攻击这个目标时追加出牌次数

这是一种很典型的本体实现思路：

- 卡牌负责“打标签/挂状态”
- Power 负责“监听后续条件并放大效果”

联机协同效果很多都是这样写的，而不是全塞进卡牌 `OnPlay(...)` 一次性完成。

## 4. 游戏本体中联机卡池是怎么处理的

### 4.1 结论

本体没有单独做一份“联机卡池类集合”。

真实做法是：

- 各角色和公共池仍然正常列出全部卡
- 发牌、奖励、商店、战斗生成时，再根据玩家人数和 `MultiplayerConstraint` 做过滤

所以联机卡池更准确地说是：

“普通卡池 + 生成阶段的多人约束过滤”

### 4.2 卡池基类

卡池基类：

- `D:\Github\spire-codex\extraction\raw\src\Core\Models\CardPoolModel.cs`

核心方法：

```csharp
public IEnumerable<CardModel> GetUnlockedCards(UnlockState unlockState, CardMultiplayerConstraint multiplayerConstraint)
```

它会先做解锁过滤，再按联机约束处理：

- `MultiplayerOnly` 模式下，移除 `SingleplayerOnly`
- `SingleplayerOnly` 模式下，移除 `MultiplayerOnly`
- `None` 模式下，不做联机方向过滤

这说明卡池模型本身就已经内建了“按多人约束筛卡”的能力。

### 4.3 各角色卡池仍然是硬列卡表

示例：

- `D:\Github\spire-codex\extraction\raw\src\Core\Models\CardPools\IroncladCardPool.cs`

结构很直接：

```csharp
protected override CardModel[] GenerateAllCards()
{
    return new CardModel[]
    {
        ModelDb.Card<Aggression>(),
        ModelDb.Card<Anger>(),
        ...
    };
}
```

也就是说，联机牌通常仍然直接放在原有角色池或公共池中，而不是另起一套“多人专用池类”。

### 4.4 真正控制联机出牌池的是 `CardFactory`

核心文件：

- `D:\Github\spire-codex\extraction\raw\src\Core\Factories\CardFactory.cs`

关键方法：

```csharp
private static IEnumerable<CardModel> FilterForPlayerCount(IRunState runState, IEnumerable<CardModel> options)
{
    if (runState.Players.Count > 1)
    {
        return options.Where(c => c.MultiplayerConstraint != CardMultiplayerConstraint.SingleplayerOnly);
    }
    return options.Where(c => c.MultiplayerConstraint != CardMultiplayerConstraint.MultiplayerOnly);
}
```

这段逻辑是理解联机卡池处理方式的关键。

意思非常明确：

- 多人局时，不允许出现 `SingleplayerOnly`
- 单人局时，不允许出现 `MultiplayerOnly`

### 4.5 哪些地方会走这层过滤

`CardFactory.FilterForPlayerCount(...)` 会被大量“生成卡”的路径复用，例如：

- 商店卡生成 `CreateForMerchant(...)`
- 奖励卡生成 `CreateForReward(...)`
- 战斗中临时生成卡 `GetForCombat(...)`
- 战斗中生成不重复卡 `GetDistinctForCombat(...)`
- 变形牌 `CreateRandomCardForTransform(...)`

因此联机卡池并不是只影响奖励牌，而是会统一影响：

- 奖励
- 商店
- 战斗内临时生成
- 变形/发现/随机获得

### 4.6 联机卡的资源加载也会跟着过滤

文件：

- `D:\Github\spire-codex\extraction\raw\src\Core\Assets\PreloadManager.cs`

在 `GetRunAssetPaths(...)` 中，若当前不是联机，会把 `MultiplayerOnly` 卡排除掉，不去预加载其资源。

这说明本体对联机牌做的不只是“抽卡阶段过滤”，还包括：

- 资源预加载阶段过滤
- 避免单机局加载不需要的联机卡资源

### 4.7 图鉴层面也有单独开关

文件：

- `D:\Github\spire-codex\extraction\raw\src\Core\Nodes\Screens\CardLibrary\NCardLibrary.cs`

图鉴界面里有 `multiplayerCardFilter`：

- 如果“查看联机卡”未勾选，则隐藏 `MultiplayerOnly`
- 勾选后才显示

这进一步说明联机卡在数据层并不是独立系统，而是普通卡牌集合中的一个受约束子集。

## 5. 对 Mod 开发的直接启发

如果要参考本体实现 Karen 的联机牌，最值得照着学的是下面这套思路。

### 5.1 效果写法仍然用普通卡牌体系

不要把“联机卡”理解成需要一套单独网络脚本。

更合适的方式是：

- 继续继承你自己的 `KarenBaseCardModel`
- 继续把效果写在 `OnPlay(...)`
- 需要持续联动时配套写 Power
- 需要跨事件响应时走 Hook 或已有管理器

### 5.2 用约束位而不是独立卡池类型区分联机牌

本体做法更偏向：

- 一张牌声明“只能单机”或“只能联机”
- 卡池保持统一
- 在生成阶段统一过滤

如果 Karen 后续要做联机专属牌，最贴近本体的方案是：

1. 卡牌仍然加入正常角色卡池
2. 给卡牌补一个类似 `MultiplayerConstraint` 的约束
3. 在奖励、商店、生成、发现等入口统一过滤

### 5.3 联机协作效果建议优先拆成“卡牌 + Power”

当效果不是“一次性结算”而是“等待队友后续动作触发”时，本体明显偏好：

- 卡牌负责挂状态
- Power 负责后续联动

这对可维护性更友好，也更容易挂到现有 Hook 流程中。

### 5.4 所有生成入口都要记得做联机过滤

如果以后 Karen 增加联机专属牌，只在奖励池做过滤是不够的。

至少要检查这些入口：

- 战斗奖励
- 商店
- 战斗内随机生成
- 变形
- 发现类效果
- 图鉴显示
- 资源预加载

否则很容易出现：

- 单机里抽到联机牌
- 联机里抽到单机限定牌
- 图鉴/资源和实际可出现卡牌不一致

## 6. 总结

游戏本体里联机卡牌并不是一套独立系统，而是在普通 `CardModel` 体系上增加“多人可用性约束”和“联机动作同步”。

从实现角度看：

- 卡牌效果仍然写在 `OnPlay(...)`
- 队友联动靠目标类型、队友遍历、Power、Hook 来完成
- 联机同步由 `ActionQueueSynchronizer` 统一处理

从卡池角度看：

- 没有专门的联机卡池大类
- 仍然是普通角色池/公共池列全量卡
- 在 `CardFactory.FilterForPlayerCount(...)` 这类生成入口统一按玩家人数过滤

对 Karen Mod 来说，这套思路非常值得复用：统一卡池、统一卡牌体系、统一生成入口过滤，而不是为联机再平行复制一整套卡牌和卡池框架。
